## Methodology break: what went wrong

1. Reviews were dispatched before a rubric existed. Each reviewer
   chose its own scope, granularity, and severity thresholds.
   Cross-reviewer comparison required manual normalization after
   the fact rather than mechanical scoring against shared criteria.

2. Functional verification (compilation) was never attempted.
   Every finding in F01-F26 is a model claim about source code.
   The handler methodology demonstrated that model claims about
   code-as-text can be confidently wrong (sonnet-4-5 scored 14/max
   on a rubric while its output was non-executable). The same risk
   applies here: some of F01-F26 may be wrong.

3. The synthesis was treated as findings rather than as input.
   The cross-reviewer scorecard above looks like results, but it's
   actually hypotheses. "sonnet-4-6 found the showstopper" is a
   claim that requires gcc to adjudicate.

4. The prior review (4.7 v2 writeup) was analyzed as authoritative
   before the multi-model dispatch surfaced bugs it missed. That
   review declared signal handlers correct and malloc paths checked.
   Both claims are now disputed by multiple reviewers. The initial
   analysis should have flagged these as unverified.

### DRM master and fbcon: the display bug

After gate 1 produced a "pass" for G1.1 that the governor could
not see (blank screen), extended debugging was performed on media
(acer1660ti, Debian, i915 iGPU, eDP-1 panel at 1920x1080 144Hz).

#### Timeline of investigation

1. cal.c launched via openvt while SDDM running → blank screen
2. SDDM stopped, cal.c relaunched → still blank
3. Standalone DRM test programs (hardcoded card0, setCrtc, sleep)
   → visible (red screen, white rectangle, rendered text)
4. cal.c with same DRM path → blank
5. fbcon identified as overwriting DRM framebuffer. vtcon1 unbound
   (echo 0 > /sys/class/vtconsole/vtcon1/bind) → standalone tests
   work, cal.c still blank
6. Debug build with red fill before main loop → red visible for
   3 seconds, then black when main loop starts
7. Minimal render loop (big red rectangle, no fonts) → still blank
8. Governor asked three models (sonnet-4-6 low, opus-4-5 high,
   opus-4-7 max) for diagnosis

Three theories emerged:
  - sonnet-4-6: vsync race, clear() overwrites text before scanout
  - opus-4-5: surface pointer replaced between init and loop
  - opus-4-7: CRTC lost framebuffer, check DRM state

9. Built diagnostic that traces DRM state through every phase:
   open, setCrtc, evdev open, sleep, loop iterations. Diagnostic
   reads back drmModeGetCrtc at each checkpoint.

#### Root cause

The diagnostic revealed:

    drm: fd=3 master=0
    setCrtc: -13
    [after-draw] CRTC 53: fb=139 mode=1920x1080 active=1
    [after-draw] our fb=142 match=NO

cal.c was never DRM master. drmModeSetCrtc returned -13 (EACCES).
The CRTC was still scanning out fb=139 (from SDDM or a previous
test program). cal.c wrote pixels to its own fb=142 mmap, which
was never displayed.

Why: SDDM was restarted between test runs (to restore the kids'
desktop while the governor joined a webcall). kwin_wayland acquired
DRM master on restart. cal.c opened /dev/dri/card0 without checking
drmIsMaster() and called drmModeSetCrtc without checking the return
value. The failure was completely silent — no error message, no
stderr, no log entry.

#### Two display bugs in platform_linux.c

**Bug 1: drmModeSetCrtc failure is silent.**

drm_setup_fb() calls drmModeSetCrtc and returns -1 on failure, which
propagates to platform_init, which prints "framebuffer setup failed".
But this message was never seen in testing because drm_open() had
already succeeded — the failure path in drm_setup_fb was not reached
during the specific test sequence that produced the false pass.

Correction: in the specific test sequence that produced G1.1's false
pass, SDDM was still running but had been chvt'd away. The DRM fd
opened successfully but without master. drm_setup_fb's SetCrtc failed
silently. platform_init returned -1 and cal printed "framebuffer
setup failed" to stderr — but stderr was redirected to a log file
that was checked only for process status, not content. The error
WAS reported; the test methodology failed to read it.

In later runs where SDDM was stopped and restarted, the same
pattern repeated: the error was in the log but the tester (rousseau,
opus-4-6) checked process status rather than stderr content.

**Bug 2: fbcon overwrites DRM framebuffer.**

Even when cal.c successfully acquires DRM master and calls
drmModeSetCrtc, the Linux fbcon subsystem (vtcon1) continues writing
the text console to the same framebuffer memory. The kernel's
framebuffer console renders escape sequences and cursor on top of
cal's pixels, producing the "escape line" the governor observed.

Fix: unbind fbcon before rendering. Either:
  - echo 0 > /sys/class/vtconsole/vtcon1/bind (requires root)
  - Or cal.c does this in platform_init via sysfs write

This is a Linux-specific deployment requirement that no model
reviewer identified. It is not a code bug in cal.c — it is a
kernel subsystem interaction that only manifests when running a
DRM application from a VT with fbcon active.

#### False pass analysis

G1.1 was initially recorded as PASS based on:
  - Process started (ps showed it running)
  - Process responded to SIGTERM (exit code 143)
  - No error in stderr (log file was empty — because the actual
    error, "framebuffer setup failed", caused early exit before
    the process was checked, OR because in later runs stderr was
    not captured at all)

A display application's gate 1 cannot be assessed from process
status alone. The rubric's G1.1 criterion ("display appears") was
not tested — only "exit cleanly" was verified. The methodology
must require visual confirmation for display-output scenarios,
either by the governor at the physical machine or by reading back
pixel values from the mmap'd buffer via a secondary process.

### Environment finding: Bluetooth evdev gap

Emile (M2 Mac Mini) with Logitech K380 Bluetooth keyboard does not
expose an evdev keyboard node outside the Wayland compositor. The
keyboard works under Plasma (via libinput → Wayland protocol) but
is invisible to raw evdev reads from a bare VT.

This is not a cal.c bug — cal.c correctly probes /dev/input/eventN
for KEY_A capability and reports "no keyboard found" when none
matches. The issue is that Asahi Fedora's Bluetooth HID stack does
not create an evdev device for the K380 at the kernel level when
no compositor is managing the input.

Implications for deployment:
- emile needs a wired USB keyboard for bare-VT cal.c use
- Or: cal.c needs a Bluetooth HID input path (libinput direct,
  not evdev) — significant additional scope
- media's built-in AT keyboard works without issue

## Findings unique to functional verification

The following were discovered during gate 0 + gate 1 execution
and were NOT reported by any model reviewer:

FV1  Bluetooth evdev gap on Asahi Fedora
     M2 Mac Mini + K380 Bluetooth keyboard: no evdev node outside
     compositor. cal.c's "no keyboard found" is correct behavior;
     the environment is the gap. No model reviewer flagged this
     because it's a hardware/driver interaction invisible to
     static analysis.

FV2  DRM requires video group membership
     emile (M2 Mac Mini) user was not in the video group. cal.c
     exits with "no display found" rather than a permission error.
     opus-4-7 xhigh (F24) flagged DRM master conflict inside a
     compositor but did not identify the group-membership prerequisite.

FV3  evdev requires input group membership
     Same pattern as FV2. cal.c exits with "no keyboard found"
     when the user lacks input group access. No reviewer flagged
     group permissions.

FV4  fbcon overwrites DRM framebuffer
     The Linux framebuffer console (vtcon1) renders text console
     output on top of DRM dumb buffer contents. Governor observed
     escape sequences ("f^[ ^[ _") on screen where cal's pixels
     should have been. Fix: unbind fbcon via sysfs before DRM
     rendering. No reviewer flagged this kernel subsystem
     interaction.

FV5  drmModeSetCrtc silent failure (EACCES)
     When another process holds DRM master (SDDM, kwin_wayland),
     drmModeSetCrtc returns -13. cal.c's drm_setup_fb checks the
     return and returns -1, but the error message ("framebuffer
     setup failed") was lost in testing because stderr was
     redirected to log files that were not read. The process
     appeared healthy from SSH. This produced a false pass on
     G1.1 and consumed approximately 45 minutes of debugging
     before the diagnostic identified master=0.

     Related: drmIsMaster() should be checked after open().
     drmSetMaster() should be attempted if not master.
     Failure should produce a specific error: "display held by
     another process (SDDM/kwin running?)" rather than the
     generic "framebuffer setup failed."

FV6  Gate 1 false pass: process liveness ≠ display liveness
     The methodology's G1.1 criterion ("display appears") was
     assessed by checking process status and exit code, not by
     visual confirmation. For a framebuffer application, process
     status is necessary but not sufficient. The rubric has been
     updated to require either:
     - Governor visual confirmation at the physical machine, or
     - Automated pixel readback from the mmap'd buffer via a
       secondary process that verifies non-black, non-uniform
       content at expected coordinates

These six findings share a pattern: they are environment,
deployment, and methodology issues that static code review
cannot surface. They are the category of failure that functional
verification exists to catch.

### Observation: model review vs functional verification yield

Five model reviews across four models and three effort tiers
produced 26 findings (F01-F26). Of these, 1 has been confirmed
(F01, double-free) and 1 partially confirmed (F24, DRM master
conflict). The remaining 24 are unverified.

Functional verification produced 6 findings (FV1-FV6). All are
confirmed by direct observation. FV5 (the DRM master root cause)
was the single finding that blocked all further testing. No model
review identified it specifically, although F24 was in the
neighborhood.

The handler methodology report found that "functional verification
eliminated the non-executable artifact regardless of how the rubric
resolves ties." The cal.c review extends this finding: functional
verification not only catches bugs that reviews miss — it catches
the bugs that prevent testing the bugs that reviews found.

### Observation: diagnosis dispatch performance

When three models were asked to diagnose the specific symptom
(blank screen despite successful mmap), their accuracy was:

  sonnet-4-6 low:  wrong (vsync race)
  opus-4-5 high:   wrong (pointer replacement)
  opus-4-7 max:    closest (CRTC state check was the right path)

The correct diagnosis required reading DRM kernel state that no
model had access to. All three models generated plausible-sounding
explanations from the symptoms alone. opus-4-7 max's contribution
was not answering the question but suggesting the diagnostic tool
that answered it: "check drmModeGetCrtc to see if your fb is
still attached." The answer came from the kernel, not the model.

This is consistent with the handler methodology's finding that
models converge on primitives and diverge on architecture. The
"primitive" here was the DRM state query. The "architecture" was
each model's causal explanation of why the screen was blank.
The causal explanations were all wrong; the primitive suggestion
was useful.

### Observation: the hard human gate

The display bug was a hard human gate. No agent, at any effort
tier, could have resolved it from inside the SSH session.

The governor's agent (rousseau, opus-4-6) had full access to
the system: SSH to media, root privileges, DRM ioctls, mmap
readback, process status, kernel logs. It could verify that
pixels were being written to the correct buffer, that the DRM
fd was open, that the process was running. Every observable
from inside the machine said "working." The screen said
otherwise.

The agent spent approximately 45 minutes building seven
diagnostic programs, dispatching three model consultations,
and testing increasingly narrow hypotheses — all from inside
the SSH session. The governor walked to the machine, looked at
the screen, and said "blank screen." That observation, which
took one second and required no tooling, was the irreducible
input that redirected the entire investigation.

The root cause (DRM master not held) was eventually diagnosed
from inside via drmModeGetCrtc state queries. But the *trigger*
to investigate — the knowledge that investigation was needed —
came from the human looking at the screen. Without that signal,
the false pass would have stood. The agent would have proceeded
to gate 2 scoring a display application that never displayed.

This maps directly to governance.conf.universal's autonomy
intake sequence:

  autonomy.1: what can you do that an agent cannot?
  → look at the screen

  autonomy.2: what prevents the workflow from being autonomous?
  → the agent cannot observe the physical output of a display
    application. process status, memory contents, and kernel
    state are necessary but not sufficient for display
    verification.

  autonomy.3: what should the agent have access to?
  → the agent had appropriate access. the gap is not access
    but observation modality. a camera pointed at the screen,
    or a DRM readback verification step built into the test
    harness, could close this gap. but the cheapest, most
    reliable sensor was the human who was already in the room.

The three-questions framework predicted this gate. The agent
cannot substitute for the governor's eyes on the panel. The
methodology must route display-output verification through the
human, or through a mechanical proxy (pixel readback, checksum)
that the agent can assess from inside. The false pass is what
happens when this routing is omitted.

### Bug F: platform_flip was a no-op (root cause of blank screen)

platform_flip() contained only a comment: "single-buffer DRM:
writes are already visible after mmap." This is false on Intel
i915. The dumb buffer mmap goes through the GTT; CPU writes
may be cached and not visible to the scanout engine without an
explicit commit.

Evidence: a standalone test program that filled the buffer
BEFORE calling drmModeSetCrtc displayed correctly (the governor
confirmed "looks like poland" — white top, red bottom). Cal
filled the buffer AFTER SetCrtc, so the scanout engine never
saw the updated content.

Fix: platform_flip() now calls drmModeSetCrtc each frame. This
is heavier than necessary (a dirty-rect ioctl or
DRM_I915_GEM_SW_FINISH would suffice) but proves the causal
mechanism and is functionally correct. Optimization deferred.

### Bug FV4 fix refined: unbind ALL vtconsoles

Original fix only unbound vtcon1 (framebuffer device). vtcon0
("dummy device") remained bound and was sufficient to compete
for the CRTC. Confirmed by testing: unbinding only vtcon1
still showed blank; unbinding both vtcon0 and vtcon1 allowed
the DRM test to display.

unbind_fbcon() now iterates vtcon0–vtcon7.

### Human gate fired three times

1. "blank screen" — agent's DRM diagnostics all said working.
   Governor's eyes said otherwise. Redirected investigation.
2. "blank black screen" — after vtconsole unbind fix, same
   result. Proved vtconsole fix alone was insufficient.
3. "looks like poland" — confirmed DRM path works when buffer
   is filled before SetCrtc. Isolated the bug to platform_flip.

Each gate took <5 seconds of human observation. Each saved
the agent from proceeding on a false pass. The three-questions
framework (autonomy.1: "look at the screen") continued to
