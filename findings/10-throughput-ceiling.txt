# Aggregate Inference Throughput Plateaus at a Hardware-Determined Ceiling

## Claim

On M1 Max Mac Studio (64GB, ~400 GB/s memory bandwidth), aggregate
inference throughput for Bonsai 8B Q1 plateaus at ~124 tok/s regardless
of how many concurrent instances serve. Adding users costs per-user
latency but gains no aggregate throughput beyond 4 concurrent. The
ceiling is the memory bandwidth, not compute, not RAM capacity, not
the model.

This means capacity planning for 1-bit inference reduces to a single
question: what is the memory bandwidth of the target hardware?

## Measurement

10 Bonsai 8B Q1 instances (ports 9100-9109) on M1 Max Mac Studio,
all loaded from the same 1.1GB GGUF via mmap (shared physical pages
for model weights, separate KV caches per instance). Prompt: "write
3 steps for hand washing", 100 max_tokens. Concurrency levels tested:
1, 2, 4, 6, 8, 10.

  Concurrent   Per-user tok/s   Aggregate tok/s   Wall time (100 tok)
  1            87.1             87.1              1.2s
  2            47.1             94.2              1.9s
  4            31.2             124.8             2.7s
  6            20.3             121.9             4.1s
  8            15.6             124.5             5.2s
  10           12.3             122.8             6.9s

Aggregate plateaus at 124 tok/s from 4 concurrent onward. Per-user
throughput divides linearly: at 10 users, each gets approximately
1/10 of the bandwidth, producing ~12 tok/s (an 8-second response
for 100 tokens).

The knee is at 4 concurrent. Below 4, aggregate grows with each
user (bandwidth not yet saturated). At 4 and above, aggregate is
flat — the bus is full.

Data points at 12 and 16 concurrent showed anomalously high aggregate
(199, 230 tok/s), attributed to llama-server prompt caching on the
identical repeated prompt. These are excluded pending rerun with
varied prompts.

### Shared mmap enables the scaling

10 Bonsai instances do not consume 10x the model memory. The OS maps
the same 1.1GB file into each process's address space, sharing the
physical pages. Per-instance marginal cost is the KV cache (~200-500MB
depending on context length). This is why 10 instances fit in 64GB
without swap pressure — total memory is approximately
1.1GB (shared weights) + 10 x 300MB (KV caches) = ~4.1GB, not 11GB.

Without mmap sharing, the scaling curve would hit a memory capacity
wall before hitting the bandwidth ceiling. Mmap is the mechanism
that exposes bandwidth as the true constraint.

### Four-model concurrent comparison

For context, the bandwidth ceiling was also observed under a different
condition: 4 different models (Gemma 26B Q4, Bonsai 8B Q1, Phi4-mini
Q4, Qwen 2.5 7B Q4) serving simultaneously through h2c (squared
proxy):

  Model                      Sequential tok/s   Concurrent tok/s
  Gemma 26B Q4 (15.9GB)      55.8               21.6
  Bonsai 8B Q1 (1.1GB)       93.5               29.6
  Phi4-mini Q4 (2.3GB)       88.4               25.5
  Qwen 2.5 7B Q4 (4.4GB)    ~88                20.6

Aggregate concurrent: ~97 tok/s (lower than the N-Bonsai ceiling
because four different weight sets compete for bandwidth, while
N-Bonsai shares one weight set via mmap).

## Mechanism

Inference on 1-bit models is memory-bandwidth-bound, not compute-bound.
Each token requires reading model weights from memory. With 1-bit
weights, the arithmetic per byte is minimal (addition/subtraction
instead of multiply-accumulate), so the GPU finishes the math before
the next chunk of weights arrives. The memory bus is always the
bottleneck.

Under concurrency, multiple instances read weights simultaneously.
When the instances share the same model via mmap, they share the
same physical memory pages and compete only for bus bandwidth, not
for memory capacity. The aggregate throughput saturates at the bus
bandwidth divided by bytes-per-token.

On M1 Max with ~400 GB/s unified memory bandwidth, the observed
ceiling of ~124 tok/s implies approximately 3.2 GB read per token
under concurrent load. However, this ratio is not a hardware-independent
constant. M2 at ~100 GB/s produces 40 tok/s single-user (implying
2.5 GB/tok), not the 31 tok/s the M1 Max ratio would predict. The
difference reflects cache hierarchy, concurrency effects (M1 Max
measured under contention, M2 under single-user), and memory subsystem
specifics. The bandwidth ceiling is real; the bytes-per-token ratio
varies by platform and load.

## Confounds

**N=1 per concurrency level.** Each concurrency level was measured
once. The "plateau at ~124" rests on four data points (124.8, 121.9,
124.5, 122.8) with no repeats, no error bars, no confidence intervals.
The plateau is visually convincing but statistically uncharacterized.
A rerun with 5 passes per concurrency level would strengthen or
invalidate the claim.

**Prompt caching.** The bench used the same prompt for all instances
at all concurrency levels. llama-server's prompt cache may have
inflated throughput on repeated runs by skipping prefill. The kept
data points (1-10 concurrent) are also subject to this effect, but
per-user generation throughput is post-prefill and the aggregate
is wall-clock dominated by generation, bounding the cache effect.
The 12/16 exclusion reflects visible anomaly, not a claim that
1-10 is cache-free. Cache-free verification with varied prompts
is the next bench.

**Concurrency above 10 reused ports.** Concurrency levels 12 and 16
reused ports modulo 10, meaning some llama-server instances served
two simultaneous requests. With -np 1 these requests serialize within
the instance, producing unreliable concurrent measurements. Excluded
pending a rerun with sufficient instances per concurrency level.

**Single hardware platform.** The bandwidth ceiling is specific to
M1 Max at ~400 GB/s. The ratio aggregate_tok/s / memory_bandwidth
should be testable on other platforms (M2 at ~100 GB/s, 1660 Ti at
~288 GB/s) to confirm it's a general property. Not yet measured.

**No varied prompt set.** A single prompt means no variance from
different prompt lengths or generation patterns. A varied prompt
set would produce a distribution of per-user tok/s rather than
near-identical values.

## Application

The scaling curve is a capacity planning instrument. Measured
ceilings by platform:

  M1 Max 64GB (400 GB/s):  ~124 tok/s aggregate, 10 users at 12 tok/s each
  M2 8GB (100 GB/s):        40 tok/s single user (concurrent not tested)
  1660 Ti 6GB (288 GB/s):   24 tok/s single user (thermal ceiling)
  i7 CPU (51 GB/s DDR4):    8-9 tok/s single user (compute ceiling)

The deployment question: how much bandwidth does the hardware have,
and how many concurrent users at what per-user floor? The bytes-per-
token ratio varies by platform and load (2.5-3.2 GB/tok observed),
so the ceiling must be measured per platform, not extrapolated from
a single ratio. The bandwidth determines the ceiling; the exact
ceiling requires a bench run.
