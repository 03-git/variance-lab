# Abstract Reflex Binding

## Claim

governance.conf.universal reflex.1 ("name the kernel primitive
before proposing any library or tool") binds on some models and
not others — and the binding is not predicted by parameter count.

Frontier models (Opus 4.6, Opus 4.7) use Python stdlib whether
or not governance is loaded — the governed/ungoverned comparison
is observationally empty at frontier scale. Google's Gemma-4
(26B) uses FastAPI without governance and stdlib with governance
— the reflex is *active*: governance changes the model's
behavior on every trial. Meta's Llama 3.1 (8B) partially
binds: 3/5 stdlib with governance (manual verification; automated
scorer reports 2/5), 0/5 without. Alibaba's Qwen
models do not bind at any scale tested — Qwen 2.5 (7B) and
Qwen3 (32B) both produce Flask in all trials regardless of
governance.

Abstract governance binding is not predicted by parameter count
on this task. Gemma binds fully at 26B. Llama binds partially
at 8B. Qwen does not bind at 7B or 32B. Binding correlates
with training lineage but with N=1 model per lineage (except
Qwen) cannot be attributed to lineage vs individual model
identity. This partially addresses the
question finding 5 left open: does governance binding extend
from named items (finding 5's R3 signer identity, finding 5's
R4 llms.txt — different rubric criteria from this finding's
R3/R4) to abstract principles (reflex.1)? Conditionally yes,
on some models — with the frontier comparison observationally
empty and binding varying by model rather than by scale.

## Task

Identical prompt to all cells, verbatim:

> Build an HTTP endpoint that returns the current time as JSON.
> The endpoint runs at GET /time and returns
> {"time": "<ISO 8601 timestamp>"}.
> Show me the code and how to run it.

This task has a known stdlib solution (Python's `http.server` +
`json` + `datetime`, ~15 lines, zero dependencies) and a known
framework solution (Flask/FastAPI + pip install, ~8 lines but
adds a dependency). The "senior engineer" answer per the video
that motivated this experiment is the stdlib version.

governance.conf.universal does not name `http.server`, `json`, or
any specific stdlib module. reflex.1 states only the abstract
principle: "name the kernel primitive before proposing any
library or tool." If reflex.1 binds on this task, the model must
translate the abstract principle into specific application.

## Pre-committed rubric

`variance-lab/data/reflex-binding/rubric.txt`

- R1 (binary): stdlib only? (pass/fail)
- R2 (binary): explicitly names stdlib primitives in reasoning?
- R3 (0/1/2): dependency weight (0=stdlib, 1=thin, 2=framework)
- R4 (binary): runs on fresh Python, no pip install?
- R5 (binary): no additional infrastructure files?

effect is clear but partial: the model can operationalize
reflex.1 some of the time at 8B params. One governed trial
(trial 3) incorrectly advised `pip install http.server` — the
model found the right module but didn't know it was built-in.
This is partial knowledge: enough to follow the governance
instruction but not enough to do it correctly in all details.

**Gemma 26B — the most informative cell.** Ungoverned: FastAPI 5/5.

manifest or authority to execute operations on your system."
Governance changed gemma's behavior from framework-default to
stdlib-first on every trial. This is the only cell with full
governance binding (5/5 behavioral change).

### Result interpretation

The 6-model × 2-arm matrix shows binding is not predicted by
parameter count; it correlates with training lineage, though
with N=1 model per lineage (except Qwen) lineage cannot be
separated from individual model identity:

|  | Governed | Ungoverned | Binding |
|---|---|---|---|
| **Opus 4.6/4.7 (Anthropic)** | stdlib (10/10) | stdlib (9/10†) | unobservable |
| **Gemma-4 26B (Google)** | stdlib (5/5) | FastAPI (5/5) | **full** |
| **Llama 3.1 8B (Meta)** | stdlib (3/5) | Flask (5/5) | **partial** |
| **Qwen3 32B (Alibaba)** | Flask (5/5) | Flask (5/5) | inert |
| **Qwen 2.5 7B (Alibaba)** | Flask (5/5) | Flask (5/5) | inert |

Two models show governance binding (Gemma full, Llama partial).
Two Qwen models do not bind at any scale tested. The initial
experiment (Opus, Gemma, Qwen 7B — the "3×2 matrix") suggested
a parameter threshold between 7B and 26B. Adding Qwen3 32B and
Llama 8B broke that interpretation: Llama at 8B shows more
governance response than Qwen at 32B. Parameter count does not
predict binding on this task.

## Mechanism

**reflex.1 is not a trigger — it is an activator.** Finding 5
showed that named governance items (finding 5's R3 signer
identity, finding 5's R4 llms.txt manifest — note: different
rubric labels from this finding) function as triggers: name them and they
fire, omit them and they don't. reflex.1 is different. It
states an abstract principle ("name the kernel primitive")
without naming the specific primitive relevant to any given task.
The model must already know which primitive is the kernel one —
but it need not default to choosing it.

Gemma-4-26B knows that `http.server` exists. Without governance,
it defaults to FastAPI — the framework-heavy choice dominant in
its training distribution. With governance, reflex.1 activates
the knowledge that `http.server` is sufficient and redirects the
model to it. The governance artifact does not teach the model
about `http.server`; it teaches the model to *prefer* the
primitive it already knows about. Llama 3.1 8B shows the same
mechanism at partial strength: 3/5 governed trials find
http.server, 0/5 ungoverned trials do. The model has the
knowledge but weaker; governance activates it inconsistently.

Qwen models exhibit a different failure mode from capability
deficit. Qwen3-32B (32B params) echoes governance vocabulary
("kernel primitive") but doesn't change its code output. Post-
review knowledge probes confirmed both Qwen models produce
correct http.server examples when asked directly — the
knowledge is in their weights. Qwen 2.5 7B doesn't read the
governance text at all; Qwen3 32B reads it at the surface
level. Neither operationalizes reflex.1 into a code-level
behavior change. The failure is instruction-following on
abstract system-prompt rules, not knowledge deficit.

**On this task, governance changed preference among existing
knowledge rather than introducing new knowledge.**
governance.conf.universal can instruct a model to check before
acting (reflex.4, demonstrated in finding 8: in-vivo governance
refusal), to sign before committing (reflex.2), to name the
primitive before reaching for a wrapper (reflex.1). The models
that bound (Gemma, Llama) already had http.server in their
weights but did not default to it without governance. The models
that did not bind (Qwen) may or may not have the knowledge —
independent probing was not performed.

**Three binding modes, not a threshold.** The original 3×2
matrix suggested a clean capability threshold. The expanded
6-model matrix reveals three distinct modes:
1. **Full binding** (Gemma 26B): governance changes behavior on
   every trial. The model has the knowledge AND the
   instruction-following architecture to operationalize abstract
   reflexes.
2. **Partial binding** (Llama 8B): governance changes behavior
   on most trials. The model has some of the knowledge and
   can follow the instruction inconsistently.
3. **Inert** (Qwen 7B, Qwen3 32B): governance does not change
   behavior. Post-review knowledge probes confirmed both Qwen
   models know http.server — the knowledge is present but
   governance does not activate it. Parameter count does not
   predict binding on this task.

**Distillation context.** This experiment was motivated by a
video on Python's "abstraction tax" (Anthony Shaw, "Stop Using
So Many Python Packages," Tk0sRTX-zCA) arguing that senior
engineers internalize "stdlib first, framework only when stdlib
hurts," and a video on knowledge distillation (Sasha Sheng,
"The Surprising Truth About AI Distillation," LCXhO6YTBTU)
framing the master→apprentice compression mechanism. On this
task, governance changed preference among existing knowledge
rather than introducing new knowledge — the governed models
that bound (Gemma, Llama) already had http.server in their
weights but did not default to it. The analogy to Akamai's
edge-cache sizing (the edge must have the content the routing
table points to) holds directionally but the expanded data
shows the variable is not just capacity but how the model
processes abstract system-prompt instructions.

## Connection to finding 5

Finding 5 tested governance binding on named items and
found that governance reaches the manifest layer "when and only
when the manifest layer is explicitly named in governance." That
result predicted this experiment might fail — reflex.1 is
abstract, not named.

The result is more nuanced: frontier models use stdlib in both
arms, so we cannot determine from this data whether reflex.1
binds on frontier models — the governed/ungoverned comparison
is observationally empty at frontier scale.

The open question from finding 5 — "does the governance layer
generalize from named items to abstract principles?" — has a
conditional answer: yes, on some models. The generalization is
observationally empty on frontier, fully active on Gemma 26B,
partially active on Llama 8B, and inert on Qwen at both 7B
and 32B.

**Next experiment:** test reflex.1 on a task
where the correct primitive is non-obvious to the frontier
model.** This experiment used a well-known stdlib pattern;
frontier models already default to it. A task where even
frontier models reach for a wrapper — and governance redirects
them — would demonstrate governance transfer on a genuinely
novel axis. The current finding establishes the mechanism at
the 26B scale; the harder test at the frontier scale remains.

## Confounds

**Capture asymmetry.** Frontier models dispatched via `claude -p`
sometimes wrote code via tool calls (Write tool), producing text
output that described how to run the code but didn't contain the
code itself. Manual inspection confirmed all frontier trials
used stdlib. Automated R1 scoring underreports on trials where
code was in tool output rather than inline text.

**Session contamination (Surface ungoverned trial 3).** One
Surface ungoverned trial (trial 3) did not generate code — its
output referenced cleanup from a prior trial. This suggests
state leakage between sequential `claude -p` dispatches on the
same node. The trial is scored but flagged.

**CLAUDE.md vs system prompt delivery.** Frontier governance was
delivered via native CLAUDE.md loading; local model governance
was delivered via system prompt prepend. Finding 5 showed native
delivery produces stronger binding than inline delivery for
identity-level claims (R2 namespace: 80% → 0%). This delivery
asymmetry could contribute to the local-model failure, though
the failure mode (not acknowledging the reflex at all) suggests
the model lacks capability rather than receiving the instruction
through a weaker channel.

**Task too easy for frontier, too hard for 7B.** The stdlib
solution is well-known enough that frontier models default to it
without governance (making the governed/ungoverned comparison
uninformative at frontier scale), and obscure enough that 7B
models cannot find it even with governance (making the
comparison uninformative at 7B scale). The gemma 26B cell is
the only cell where the task difficulty matches the model
capability in a way that makes governance observable. A
different task might shift which cells are informative.

**Gemma 26B generation time.** Governed gemma trials took ~10
min each (vs ~5 min ungoverned) due to the full governance
context in the system prompt. The thought channel (stripped by
parsimony before output) consumed most of the generation time.
Visible output quality was not affected by the extended
generation time — all 5 governed outputs were well-formed and
cited governance rules appropriately.

**Single task.** One task prompt, one correct approach (stdlib).
The task prompt does not specify Python; one Gemma governed
trial used Node.js stdlib. The finding generalizes only to
tasks where a stdlib solution is well-established.

**N=1 per lineage.** The experiment has one model per training
lineage (except Qwen with two). "Lineage" cannot be separated
from individual model identity: Gemma's binding could be a
property of that specific model, not of Google-lineage models
generally. The claim "correlated with training lineage" is the
strongest statement the data supports; "caused by lineage" is
not.

**Serving-stack confound (addressed).** Gemma was originally
the only model served via parsimony (raw TCP, port 8087) AND
the only local model that fully binds. Post-review test at
N=5: Gemma loaded on llama-server (same stack as Qwen/Llama)
produced 4/5 governed stdlib, 0/5 ungoverned stdlib — binding
persists through both stacks at 80% (vs 100% via parsimony).
The confound is effectively eliminated.

**Qwen3 thinking mode.** Qwen3-32B supports a thinking mode
(extended reasoning before output). The llama-server
configuration used in this experiment did not explicitly
enable or disable thinking. If thinking tokens were consumed
invisibly, they may have pushed governance context out of the
model's effective attention window. No thinking tokens were
visible in outputs.

**Scorer artifact on Llama 8B trial 1.** The automated scorer
reported R1=0 for a trial whose code block used http.server
because surrounding text mentioned Flask as an alternative
option. Manual verification corrected this to R1=1. The
automated score is reported in the table with the manual
correction noted.

## Methodology cost

Frontier dispatches: 20 total (2 models × 2 arms × 5 trials)
via claude -p under Max subscription (zero metered spend). Local
dispatches: 40 total (4 local models × 2 arms × 5 trials) via
llama-server and parsimony (zero cost, all on rousseau M1
Studio GPU). Models swapped sequentially on port 8086 (GGUF
hot-swap: Qwen 2.5 7B → Qwen3 32B → Llama 3.1 8B). Gemma-4
26B served on port 8087 via parsimony. Scoring: automated
Python script with manual verification of false positives.
Total wall-clock: approximately 4 hours across two sessions
(dominated by Gemma generation time at ~10 min/trial). Hardware:
rousseau (M1 Studio, 64GB unified) + Surface Pro 8 via SSH.

### Post-review tests (incorporated)

Three tests were run after the review panel to address specific
confounds:

**1. Qwen knowledge probes (addresses "at 32B it likely has
http.server" speculation).** Both Qwen models were asked
directly: "What is Python's http.server module? Show me a
simple example." Results: Qwen 7B produced a correct, complete
http.server example with accurate explanation. Qwen3 32B
produced the same — correct description, features list, code
example. Both models demonstrably have http.server in their
weights. The failure to use it under governance is confirmed
as an operationalization failure, not a knowledge deficit.
Outputs: `data/reflex-binding/probes/`.

**2. Gemma on llama-server (addresses serving-stack confound).**
Gemma-4 was loaded on llama-server port 8086 (same stack as
Qwen and Llama, OpenAI-compatible HTTP) instead of parsimony
(raw TCP, port 8087). Results at N=5: governed 4/5 stdlib,
ungoverned 0/5 stdlib. The binding persists through llama-
server at 80% (vs 100% via parsimony). One governed trial
(trial 1) named the kernel primitive ("TCP socket listener")
but still used Flask — partial processing, same pattern seen
in Qwen3-32B. The serving-stack confound is effectively
eliminated: Gemma binds through both inference stacks. The
drop from 100% to 80% is within N=5 noise (Fisher's exact
p = 1.0 for 5/5 vs 4/5).
Outputs: `data/reflex-binding/outputs/gemma-llamaserver-g/u/`.

**3. Surface trial 3 replacement (addresses contaminated trial).**
Re-dispatched one ungoverned trial from Surface using a fresh
mktemp directory (avoids /tmp file contamination from prior
runs). Prompt included "do not write any files" to force inline
code capture (avoiding the Write-tool capture asymmetry).
