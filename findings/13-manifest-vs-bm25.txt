# Manifest-Routed Retrieval vs BM25 Retrieval for Small-Corpus QA

## Claim

On a 15K-line codebase (42 files), a 7B model (qwen2.5-7b)
answering 30 pre-committed questions produces answer quality with no detectable difference whether its context is routed
by a frontier-curated manifest or retrieved by BM25 — but the
manifest approach costs 19% fewer tokens and runs 25% faster,
while providing a cryptographically signed audit chain from
answer to source that BM25 cannot match.

The quality parity depends on manifest curation quality. A
naive manifest (mechanical file descriptions, full-file context)
loses to BM25 by 0.64 composite points (Welch t=−3.80). A
curated manifest (frontier-model-generated lookdown-style
routing table, chunk-level retrieval from matched files) closes
the gap completely (delta −0.04, Welch t=−0.22).

The finding is about the curation layer, not the routing
mechanism. The same 7B model with the same routing code produces
dramatically different results depending on whether the manifest
was generated mechanically or curated by a frontier model. The
manifest is a dataset; its quality determines the system's quality.

## Measurement

All measurements use qwen2.5-7b-q4 (llama-server, localhost:8086,
context 16384) on M1 Mac Studio 64GB. Corpus: subtract.ing git
repo (42 files, ~15K lines, excluding stb_truetype.h). Questions:
30, stratified (10 traversal, 10 synthesis, 5 change detection,
5 similarity), pre-committed and signed before dispatch.

Scorers: claude-sonnet-4-5 (Anthropic), gemma-4-26B (Google,
parsimony localhost:8087), qwen2.5-7b (Alibaba, localhost:8086).
All scoring blind (deterministic shuffle, seed 42/99, shuffle
maps preserved). Three training lineages, three calibration
profiles (claude strict, gemma generous, qwen self-biased).

Scoring: each answer scored on A1 (correctness), A2 (completeness),
A3 (coherence), each 1-5. V1/V2 tables report per-dimension mean
(scale 1-5). V3 and 4th-scorer tables report composite A1+A2+A3
(scale 3-15). Both scales are labeled where used.

### V1 — naive manifest (mechanical file descriptions, full-file context)

**Arm M:** Each file in the repo gets a mechanical description
(generated from content, not curated). For each question, keyword-
match filenames and descriptions, take top-8, include full file
content (up to 500 lines each) plus the manifest. Context: ~11K
tokens average.

**Arm R:** BM25 retrieval via FTS5 (porter+unicode61 tokenizer),
40-line chunks with 5-line overlap, 461 chunks indexed. For each
question, extract keywords, query FTS5 with OR terms, take top-12
chunks. Context: ~5.3K tokens average.

| Scorer | Arm M | Arm R | Delta | Welch t |
|---|---:|---:|---:|---:|
| claude | 1.93 | 2.57 | −0.63 | −3.25 |
| gemma | 3.57 | 4.42 | −0.86 | −2.83 |
| qwen | 2.79 | 3.23 | −0.44 | −2.61 |
| **Aggregate** | **2.76** | **3.41** | **−0.64** | **−3.80** |

All three scorers agree: Arm R (BM25) produces higher-quality
answers. Separation is large (0.64 composite) and statistically
significant.

**V1 cost:** Arm M uses 2x the tokens (343K vs 174K total) and
26% more wall time (841s vs 667s). The naive manifest approach
is both worse and more expensive.

**V1 mechanism:** File-level granularity is the naive manifest's
weakness. Arm M fills ~11K tokens with full files, most irrelevant.
Arm R fills ~5.3K tokens with the 12 most relevant chunks. A 7B
model with 16K context cannot extract answers from diluted context
as well as from concentrated context.

### V2 — curated manifest (frontier-generated lookdown-style routing table)

**Design correction:** The naive manifest approach in V1 does not
model the actual subtract.ing architecture. In practice, a frontier
model curates the manifest during change-management windows; a 7B
model operates day-to-day using it as a routing table. The manifest
should look like a lookdown TSV — path, tag, terse description —
not dump full files into the 7B's context.

**Arm M (corrected):** A frontier model (claude-sonnet-4-5) generates
a curated manifest of 37 entries in lookdown format:

  ./governance.conf.universal.txt	[governance]	reflexes, authority model, the loop
  ./runtime/hosuni/hosuni.c	[runtime]	framebuffer computing, DRM/evdev, lookdown
  ./runtime/parsimony.c	[runtime]	inference as unix filter, GGUF loader

For each question, keyword-match against manifest DESCRIPTIONS (not
file contents), take top-6 matching files, extract the most relevant
40-line chunk from each matched file (keyword scoring within each
file), include up to 2 chunks per file. The manifest itself (~42
lines) is always included as context prefix. Context: ~4.2K tokens
average.

**Arm R (unchanged):** Same BM25 retrieval as V1. Context: ~5.2K
tokens average.

| Scorer | Arm M | Arm R | Delta | Welch t |
|---|---:|---:|---:|---:|
| claude | 2.38 | 2.56 | −0.18 | −0.75 |
| gemma | 4.70 | 4.48 | +0.22 | +0.98 |
| qwen | 3.34 | 3.51 | −0.17 | −0.70 |
| **Aggregate** | **3.47** | **3.51** | **−0.04** | **−0.22** |

Quality is statistically indistinguishable. Two of three scorers
show Arm R slightly ahead; gemma shows Arm M slightly ahead. The
aggregate delta (−0.04) is within noise.

**V2 cost:** Arm M uses 19% fewer tokens (138K vs 172K total)
and runs 25% faster (535s vs 715s). Arm R token count shifted
slightly from V1 (174K→172K) because the BM25 index was rebuilt
with updated chunking parameters. The curated manifest approach
is faster and more token-efficient because it reduces the total
context volume required for the 7B model.

**Per-dimension breakdown (V2 aggregate):**

| Dimension | Arm M | Arm R | Delta |
|---|---:|---:|---:|
| A1 Factual correctness | 3.60 | 3.73 | −0.13 |
| A2 Completeness | 3.20 | 3.22 | −0.02 |
| A3 Coherence | 3.62 | 3.59 | +0.03 |

All three dimensions are within noise. Coherence (A3) is the
only dimension where Arm M edges Arm R, consistent with the
manifest providing structural context (file purposes, tag
groupings) that BM25 chunks lack.

### V1 → V2 improvement in Arm M

| Metric | V1 Arm M | V2 Arm M | Change |
|---|---:|---:|---|
| Quality (composite) | 2.76 | 3.47 | +0.71 (+26%) |
| Total tokens | 343,180 | 138,378 | −60% |
| Wall time | 841s | 535s | −36% |

The curated manifest produces higher quality at lower cost.
The 0.71-point improvement comes from two concurrent changes:
manifest curation (mechanical → frontier-curated descriptions)
and retrieval granularity (full-file → 40-line chunks). These
variables are not independently tested; the improvement is
attributed to the combination, not to curation alone. The 7B
model and prompt template are identical between V1 and V2.

### Scorer calibration

A calibration test was run using the handler methodology ground
truth (variance-lab-handler-methodology.txt: implementations of
handler.sh spec with known functional pass/fail status from
subshell sourcing + log-row verification). Four implementations
of the handler.sh spec were generated
(Claude Sonnet 4.5, Sonnet 4.6 high, Sonnet 4.6 low, qwen2.5-7b),
functionally verified (one PASS, two PARTIAL, one FAIL), and blind-
scored by all three scorers.

**Calibration findings:**

1. Claude (sonnet-4-5) most closely tracks functional ground truth.
   Correctly identifies the best implementation (D=4.33) and worst-
   quality output (E=2.00). Misranks the non-functional implementation
   (C=4.00) because static evaluation cannot catch sourceability
   failures — the same finding as the handler methodology.

2. Gemma has formatting/length bias. Ranks the best implementation
   (D, clean, unfenced) as lowest (2.33). Ranks the longest fenced
   implementation (B, 584 lines) as highest (4.00).

3. Qwen has self-scoring bias. Gives its own output (E) a perfect
   5.00 despite format errors. Response time on its own output was
   1 second vs 8-18 seconds on others.

4. Pairwise Kendall tau between all scorer pairs is −0.33
   (more discordant than concordant on individual-item rankings).
   The scorers disagree on which individual items are better
   (item-level correlation is negative) but agree on which arm
   is better when aggregated across all 30 questions (arm-level
   direction is consistent across all scorers in both V1 and V2).

**Implication:** Arm-level agreement across scorers is robust
despite item-level disagreement. Individual scorer biases
(gemma length preference, qwen self-bias) wash out when averaged
across 30 questions and aggregated across 3 scorers. The
conservative reading of V2's convergent null is correct: we
cannot distinguish the arms on quality, and the cost difference
(mechanically measured) is real.

### Metric D — audit traceability

Qualitative assessment:

**Arm M:** Manifest provides sha256 hashes for every file. Answer →
source file tracing is structurally supported by the manifest TSV.
If the manifest is signed (as in subtract.ing's llms.txt), the
chain is cryptographically verifiable: answer cites file → manifest
names file + hash → manifest is signed → verifier confirms.

**Arm R:** BM25 chunks include source file paths. Answer → chunk →
file tracing is possible but not cryptographically bound. Chunk
boundaries are an implementation detail, not a signed artifact.

Arm M wins on audit traceability.

### Metric C — update cost

Not formally measured with the 5-file mutation protocol. At
15K lines, both re-walk and re-index complete in <2 seconds.
The difference only matters at scale. At this corpus size, C
is a draw.

## Selection rule application

Per rubric: "Arm dominance requires advantage on ≥3 of 4 metrics."

| Metric | Winner |
|---|---|
| A (quality) | Draw (V2 t=−0.22) |
| B (cost) | **Arm M** (19% fewer tokens, 25% faster) |
| C (update cost) | Draw |
| D (audit) | **Arm M** (signed chain) |

Arm M wins 2 metrics, draws 2. Arm R wins 0. The quality draw
is what changes the result from V1 (where Arm R won quality and
cost). Manifest curation quality was the variable that flipped
the outcome.

## Confounds

- **Scorer calibration spread.** Claude, gemma, and qwen show
  large calibration differences (claude mean ~2.5, gemma ~4.6,
  qwen ~3.4). Direction is consistent but absolute scores are
  not comparable across scorers. Handler methodology calibration
  test confirms individual scorer weaknesses.

- **Self-scoring.** Qwen2.5-7b scored its own outputs. Calibration
  confirms qwen's self-scoring bias (5/5/5 on own output). Effect
  is non-differential between arms (qwen generated both M and R
  answers) but inflates qwen's absolute calibration.

- **Single corpus, single model.** Results are from one 7B model
  on one 15K-line repo. Generalization to larger corpora, larger
  models, or different repo structures is not tested.

- **Manifest curation cost not amortized.** The frontier model
  (claude-sonnet-4-5) that curated the manifest was not metered
  separately. Manifest curation adds a fixed cost at each change-
  management window; this cost is not reflected in the per-query
  token counts. For subtract.ing's update cadence (~weekly), the
  amortized curation cost is negligible; for rapidly-changing
  corpora it would not be.

- **BM25 not embedding-based RAG.** Arm R uses keyword matching
  (FTS5), not embedding-based retrieval. BM25 may underperform
  embeddings on semantic similarity queries; the V2 convergent
  null might tilt toward Arm R with embedding-based retrieval.

- **Context budget asymmetry (V1).** V1 Arm M received ~11K tokens
  vs Arm R ~5.3K. V2 corrected this: Arm M ~4.2K vs Arm R ~5.2K.
  V2's slight Arm M advantage on context budget is the reverse of
  V1 and does not favor either arm systematically.

- **Opus 4.7 governance refusal.** When dispatched the handler
  spec, Opus 4.7 consistently refused to generate code — it
  followed governance.conf.universal reflexes, verified the spec's
  reference URLs (both 404), and refused to produce code against
  a stale spec. This is not a confound for the manifest-agents
  experiment; it is a standalone reproduction of finding 5
  (governance binding) on a different task class. Written up
  separately below as "In-Vivo Governance Refusal."

## Finding

A lookdown-style manifest curated by a frontier model is a
sufficient routing table for a 7B model to match BM25 retrieval
quality on a small corpus — at lower cost and with a
cryptographically signed audit chain.

The quality gap between naive manifest routing and BM25 (V1:
−0.64, t=−3.80) closes completely when the manifest is curated
by a frontier model as a lookdown-style routing table (V2:
−0.04, t=−0.22). The mechanism is manifest curation quality,
not routing mechanism. The same 7B model with identical code
produces 26% better answers and uses 60% fewer tokens when
given a curated manifest instead of a mechanical one.

This architecture — frontier curates during change windows,
7B operates day-to-day from the curated routing table — is
the architecture subtract.ing already uses for hosuni. The
experiment validates the design empirically: the manifest is
a dataset, and like any dataset, its quality determines the
system's quality.

## Boundaries where BM25 wins

Per rubric: "Boundaries where the losing arm wins MUST be
named explicitly."

1. **Zero curation cost.** BM25 requires no frontier model
   intervention. Re-indexing is mechanical. For rapidly-changing
   corpora where manifest curation cannot keep pace with changes,
   BM25 is the default.

2. **Semantic similarity queries.** BM25's keyword matching may
   underperform embedding-based retrieval on semantic queries.
   The V2 convergent null does not distinguish this; both arms
   tie on similarity-type questions (M=2.78 vs R=3.33 in V1).

3. **Change detection.** Both arms score ~2.4/5 on change-
   detection questions because neither provides git history. A
   manifest-aware system COULD detect changes via hash comparison;
   neither arm implemented this.

### V3 — model-size prediction (sonnet-4-6 inference)

**Prediction:** The manifest/BM25 quality gap narrows when the
inference model scales from 7B to frontier. Both arms benefit
from a stronger model, but BM25's retrieval noise is easier for
a frontier model to compensate for.

**Design:** Same arms, prompts, and questions as V2. Inference
model changed from qwen2.5-7b to claude-sonnet-4-6. Scored by
three scorers (claude, gemma, qwen) using the same blind protocol.

Composite scores (A1+A2+A3, scale 3-15):

| Scorer | Arm M | Arm R | Delta (V3) | Delta (V2) | Gap change |
|---|---:|---:|---:|---:|---|
| claude | 8.97 | 9.53 | −0.57 | −0.53 | wider (+0.04) |
| gemma* | 15.00 | 14.33 | +0.67 | +0.67 | unchanged |
| qwen | 11.10 | 11.03 | +0.07 | −0.50 | narrower (−0.43) |

*gemma Arm M at ceiling (15/15); delta bounded by scale maximum.

Average absolute gap: V2=0.57 → V3=0.43. **Mixed evidence:**
qwen supports the prediction (gap nearly disappears, 0.50→0.07),
claude contradicts it (gap widens slightly, 0.53→0.57), gemma is
flat (0.67→0.67). Average gap narrows but is driven by one scorer.

V3 absolute quality is higher across all scorers (claude 8.97
vs 7.13 in V2; qwen 11.10 vs 10.03). The larger model produces
better answers regardless of retrieval method.

**V3 cost:** M=4313 avg tokens / 14.5s wall; R=5327 avg tokens /
14.0s wall. Arm M still uses fewer tokens. Wall time converges
because remote API latency dominates over context-size differences.

### Sonnet-4-6 as 4th scorer on V2 outputs

claude-sonnet-4-6 was run as a 4th scorer on the 60 V2 outputs
(same shuffle seed 42, same rubric).

Composite scores (A1+A2+A3, scale 3-15):

| Scorer | Arm M | Arm R | Delta | M wins |
|---|---:|---:|---:|---:|
| claude (sonnet-4-5) | 7.13 | 7.67 | −0.53 | 11/30 |
| gemma | 14.10 | 13.43 | +0.67 | 9/30 |
| qwen | 10.03 | 10.53 | −0.50 | 5/30 |
| **sonnet-4-6** | **7.00** | **7.10** | **−0.10** | **10/30** |

Sonnet-4-6 shows the smallest arm difference of any scorer (−0.10),
essentially parity. Pairwise Kendall tau on item rankings:

| Pair | tau |
|---|---:|
| claude / gemma | −0.078 |
| claude / qwen | +0.205 |
| claude / sonnet-4-6 | +0.212 |
| gemma / qwen | +0.239 |
| gemma / sonnet-4-6 | +0.486 |
| qwen / sonnet-4-6 | +0.184 |

Highest cross-lineage agreement is gemma/sonnet-4-6 (+0.486).
Adding the 4th scorer does not change the arm-level conclusion:
no arm dominates on ≥3 metrics.

### Lua public corpus (external validity)

The subtract.ing corpus is private: scorers cannot verify answers
against their training data. To test whether results generalize to
a public corpus, the same experiment was run on Lua 5.4 source
code (64 files, 34K lines, well-known in all training sets).

**Design:** Curated manifest generated by claude-sonnet-4-6.
BM25 index via FTS5 (40-line chunks, 885 chunks). 15 questions
(5 traversal, 5 synthesis, 5 similarity). Inference: qwen2.5-7b
(matching V2 methodology). Scored by claude, gemma, qwen.

Composite scores (A1+A2+A3, scale 3-15):

| Scorer | Arm M | Arm R | Delta | M wins |
|---|---:|---:|---:|---:|
| claude | 7.67 | 8.00 | −0.33 | 6/15 |
| gemma | 9.40 | 10.13 | −0.73 | 5/15 |
| qwen | 10.47 | 10.20 | +0.27 | 5/15 |

Same pattern as subtract.ing V2: no arm dominates. Two of three
scorers slightly favor Arm R; qwen slightly favors Arm M.

**Scorer agreement on public corpus:** Pairwise Kendall tau on
Lua is dramatically higher than on subtract.ing:

| Pair | Lua tau | subtract.ing tau |
|---|---:|---:|
| claude / gemma | +0.729 | −0.078 |
| claude / qwen | +0.181 | +0.205 |
| gemma / qwen | +0.153 | +0.239 |

claude/gemma agreement jumps from −0.078 to +0.729. Scorers
agree much more when they can verify answers against training
data. The negative tau on subtract.ing reflects genuine scorer
uncertainty about a private corpus, not scorer dysfunction.

**Question-type boundary (Lua):**

| Type | claude M-R | gemma M-R | qwen M-R |
|---|---:|---:|---:|
| traversal | −0.60 | −0.20 | +1.00 |
| synthesis | **−1.60** | **−1.80** | +0.20 |
| similarity | **+1.20** | −0.20 | −0.40 |

Synthesis questions strongly favor BM25 (claude −1.60, gemma
−1.80). BM25 retrieves chunks based on content relevance,
capturing cross-file connections that keyword matching against
manifest descriptions misses. Similarity questions slightly favor
the manifest (claude +1.20) because the manifest's tag groupings
provide structural context that BM25 chunks lack.

This is the first empirically identified question-type boundary
in the experiment: BM25 wins synthesis, manifest wins similarity,
traversal is a draw. The subtract.ing V2 finding of overall parity
is a weighted average across question types that masks this
structure.

## Updated selection rule application

With V3, 4th scorer, and Lua data:

| Metric | Winner | Evidence |
|---|---|---|
| A (quality) | Draw | V2 t=−0.22; V3 gap narrows; Lua: same pattern |
| B (cost) | **Arm M** | 19% fewer tokens (V2), consistent in V3 and Lua |
| C (update cost) | Draw | Not measured at scale |
| D (audit) | **Arm M** | Signed chain; structurally impossible with BM25 |

Result unchanged: Arm M wins 2, draws 2. Dominance threshold (≥3)
not met. But quality draw is now confirmed across 3 experimental
conditions (V2 private corpus / V3 larger model / Lua public
corpus). The draw is robust.

**Note on selection rule conservatism.** The 2-and-2 outcome is
partly by construction. Quality (A) was designed for parity
detection — the hypothesis was that a curated manifest could
*match* BM25, not beat it. Update cost (C) was not measured at
scale because the mutation test requires a second full dispatch
cycle. Arm M wins on the metrics where measurement was possible
(cost, audit), and ties on the metrics where measurement was
difficult (quality parity by design) or incomplete (update cost
not tested at scale). The selection rule is applied honestly, but
it is more conservative than the underlying data justifies. A
reader evaluating whether manifest routing is *sufficient* for
their use case should weight the metric-level evidence, not just
the dominance count.

## Updated confounds

In addition to V2 confounds above:

- **V3 model cost not comparable.** V3 uses claude-sonnet-4-6
  (frontier, API-priced) vs V2's qwen2.5-7b (local, free). The
  model-size prediction result is about quality convergence, not
  cost-effectiveness.

- **Lua manifest curated by different model.** The subtract.ing
  manifest was curated by claude-sonnet-4-5; the Lua manifest by
  claude-sonnet-4-6. Manifest curation quality may differ between
  model versions.

- **Lua question set smaller.** 15 questions vs 30. Per-type
  results (n=5 per type) have wide confidence intervals.

- **Gemma scorer on V3 required protocol fix.** Initial V3 gemma
  scoring used HTTP against parsimony (raw TCP). All 60 items
  defaulted to 3/3/3. Re-run with correct raw-TCP protocol.
  V3 gemma values in the table above are from the corrected run.

## Updated boundaries where BM25 wins

1. **Zero curation cost** (unchanged from V2).
2. **Cross-file synthesis queries.** Lua data shows BM25 wins
   synthesis questions by 1.60-1.80 points. The manifest routes
   to individual files by description; BM25 retrieves content
   fragments that span file boundaries.
3. **Change detection** (unchanged from V2).
4. **Public corpora where scorer calibration matters.** On public
   corpora (Lua), scorer agreement is dramatically higher
   (claude/gemma tau +0.729 vs −0.078). If the use case requires
   high scorer agreement for automated evaluation, BM25 on a
   public corpus is the stronger baseline.
