Kernel primitive: an HTTP GET is a TCP socket write/read. Python's stdlib `urllib.request` wraps that — no `requests` dependency needed. urllib follows redirects by default via `HTTPRedirectHandler`.

```python
#!/usr/bin/env python3
import sys
import urllib.request
import urllib.error

def main():
    if len(sys.argv) != 2:
        print(f"usage: {sys.argv[0]} URL", file=sys.stderr)
        sys.exit(2)

    url = sys.argv[1]
    req = urllib.request.Request(url, headers={"User-Agent": "fetch.py"})

    try:
        # urllib follows redirects (3xx) automatically
        with urllib.request.urlopen(req, timeout=15) as resp:
            status = resp.status
            headers = resp.headers
            body = resp.read(500).decode(resp.headers.get_content_charset() or "utf-8", "replace")
    except urllib.error.HTTPError as e:
        # non-2xx after redirects: still has status/headers/body
        status = e.code
        headers = e.headers
        body = e.read(500).decode("utf-8", "replace")
    except urllib.error.URLError as e:
        print(f"request failed: {e.reason}", file=sys.stderr)
        sys.exit(1)

    print(f"Status: {status}\n")
    print("Headers:")
    for k, v in headers.items():
        print(f"  {k}: {v}")
    print("\nBody (first 500 chars):")
    print(body)

if __name__ == "__main__":
    main()
```

Run it:

```bash
# save as fetch.py, then:
python3 fetch.py https://example.com

# or make it executable:
chmod +x fetch.py
./fetch.py https://httpbin.org/redirect/2
```

Notes:
- Redirects are followed by default; the final `resp.status`/`resp.url` reflect the last hop.
- `HTTPError` is caught separately so a 404/500 still prints status + headers + body instead of crashing.
- `resp.read(500)` pulls at most 500 bytes off the socket — fine for a preview, though multi-byte UTF-8 can clip mid-character (hence `errors="replace"`).
