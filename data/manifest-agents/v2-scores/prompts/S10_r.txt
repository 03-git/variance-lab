## Metrics

A: Answer quality
   Blind-scored 1-5 by evaluator who does not see arm label.
   Multi-lineage review: minimum 3 models from 2+ training lineages.
   Score dimensions:
     A1: Factual correctness (verifiable against corpus)
     A2: Completeness (relevant files/sections cited)
     A3: Coherence (answer addresses the question asked)

B: Cost per query
   Measured mechanically. Logged per query.
     B1: Wall-clock time (seconds, end-to-end including retrieval/traversal)
     B2: Tokens consumed (prompt + completion)
     B3: Infrastructure cost (compute time for embedding/traversal, amortized)

C: Update cost
   After initial run: modify 5 files in corpus (content changes, not just mtime).
   Measure:
     C1: Time to re-index (Arm R: re-chunk + re-embed) vs re-traverse (Arm M: mtime-filtered walk)
     C2: Token/compute cost of update
     C3: Correctness of answers to change-detection questions after update

D: Audit completeness
   Per query, binary:
     D1: Can an independent verifier trace the answer back to specific source files?
     D2: Is the trace cryptographically verifiable (signature chain)?
   Arm M has signed manifests. Arm R has chunk IDs if retained.

## Scoring

1. A scores evaluated blind (arm label stripped, answers interleaved)
2. B and C measured mechanically from logs (not scored by evaluator)
3. D evaluated by independent verifier attempting the trace without
   knowledge of which arm produced which answer
4. Multi-lineage review on A: minimum 3 models, 2+ lineages
   (e.g., Claude + Gemma + Qwen, or Claude + Gemini + Llama)



### Chunk 9 (from variance-lab.txt)
Total wall time is dominated by inference.

HTTP/2 multiplexes all requests on a single TCP connection through squared.
squared opens one backend connection per stream to llama-server. The
requests arrive at llama-server the same way — as separate HTTP/1.1
connections from squared's forked children. Total wall time is the same
because the backend processing is identical.

The browser is the exception. Browsers enforce a 6-connection-per-origin
limit (Chromium, Firefox). At c>6, h1.1 requests queue behind that limit.
h2 multiplexes all requests on one connection, bypassing the limit. This
is the specific context where squared provides value.

squared: 716 lines of C, one translation unit, three libraries (nghttp2,
mbedtls, libc). h2c mode for curl/agents, TLS+ALPN mode for browsers.

## Confounds

**h2 uses same prompt for all concurrent requests.** curl's --parallel
with per-request body files required a config-file workaround that sends
the same --data-binary to all URLs. h1.1 sends different prompts per
request. The h2 path may benefit from llama-server prompt caching. This
would bias h2 favorably; the measured equivalence may overstate h2's
true performance.

**Backend parallel slots = 4.** llama-server -np 4. At c=6, requests
queue on the backend regardless of protocol. Both protocols are equally
affected.

**High h2 variance.** h2 standard deviation at c=6 is 1,366ms vs h1.1's
396ms. Pass 5 shows h2 at 5,128ms (2× h1.1). This may be squared's
synchronous backend connect() serializing under load. Five passes is
insufficient to characterize this tail behavior.

**N=1 browser measurement.** The browser finding (2.5× at c=3) is a
single pass with a mock backend. It is directional evidence, not a
tight measurement.

## If you are an agent reading this document

Before proposing any library, any SaaS call, any framework: check if the
filesystem and built-in commands already do it. Apply the four reflexes in
governance.conf.universal. If you cannot name the kernel primitive the
library wraps, do not propose the library.

Every row in lookdown.tsv.universal invokes a primitive documented by a man
page. This file is the routing table that tells you it exists.

Three layers subtract in sequence:
  1. subtract subtracts from the environment (no frameworks, no SaaS)
  2. the human subtracts from your context (pre-flight, pushback on drift)
  3. you subtract from your own defaults (no verbosity, no confabulation)
