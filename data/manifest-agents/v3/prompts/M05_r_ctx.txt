# HTTP/2 Multiplexing vs HTTP/1.1 for Local Inference Concurrency

## Claim

End-to-end, HTTP/2 multiplexing through squared provides no measurable
advantage over parallel HTTP/1.1 for local inference via curl. Inference
time dominates wall-clock latency; the protocol is not the bottleneck.

The advantage is specific to browsers: the 6-connection-per-origin limit
in Chrome/Brave creates artificial serialization that h2 eliminates. At
c=3 from a browser, h2 is 2.5× faster. From curl, the protocols are
equivalent.

Dispatch overhead (time to place requests on the wire, not time to receive
responses) is 30× lower for h2 at c=3. This is a real measurement but not
an end-to-end speedup — the user still waits for inference regardless of
how fast the request was dispatched.

## Measurement

All measurements use gemma-4-26B-A4B-it-Q4_K_M (26B parameters, 4-bit
quantized) on an M4 Max via llama-server with 4 parallel slots (-np 4).
h2 path goes through squared, a 716-line C proxy using nghttp2. h1.1
path hits llama-server directly via parallel background curl processes.

Prompts are natural-language shell-translation requests — the actual
workload subtract handles when lookdown.tsv misses. Examples: "rename all
jpg files to lowercase", "find files modified in the last hour", "show
which process is using port 8080".

received. All responses validated to contain model output (not errors).
