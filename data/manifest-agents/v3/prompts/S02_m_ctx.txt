## How it renders

hosuni.c talks directly to the display hardware:

  Linux:  DRM/KMS — opens /dev/dri/card*, sets mode, double-buffered
          page flip with vsync. Reads keyboard via /dev/input/event*.

  macOS:  Cocoa — NSWindow with a backing bitmap, keyboard events via
          NSEvent. Same rendering code, different platform layer.

# hosuni.c

Framebuffer computing for humans. ~2,200 lines of C, no dependencies
beyond libc, libdrm (Linux), and Cocoa (macOS). Compiles to a single
binary under 200KB.

hosuni.c is the subtract.ing interface that replaced the browser-based
demos (demo.html, multiplying.html, the clinical pages). Those pages
required a Python bridge, a local HTTP server, and a browser. hosuni.c
needs a keyboard and a screen.

Named for the digital governess in the subtract.ing formation —
the process that sits with the kids at the framebuffer while the
governor works.

## What it does

Three modes, plus a starred-launch shortcut:

  READ  (F1)  Ask a question, get a streamed answer from the local
              inference server. Conversation history persists across
              turns. Shows time-to-first-token and tokens/sec.

  PLAY  (F2)  Launch the starred entry from the lookdown table
              directly. One keypress to resume whatever was last
              starred in RUN mode.

  RUN   (F3)  Type-to-filter launcher over the lookdown table.
              Every [player] entry in lookdown.tsv becomes a
              launchable program. Star entries for quick access
              via F2.

  TEACH (F4)  vi-subset text editor. Gap buffer, normal/insert/command
              modes, yank/paste, line numbers, ex commands (:w, :q).

White text on black background, fixed-width font rendered via
stb_truetype. The cursor is a colored block. The status bar shows
the current mode, and in READ mode, inference timing.

### Leg 3 — SOUL injection + conversation history

**Change:** ~120 lines added to runtime/cal/cal.c.
`cal_infer_stream_msgs` sends a pre-built messages JSON array
instead of a bare prompt. `build_messages_json` constructs
`[system, ...history, user]` from SOUL file + conversation
history + current query. `history_add` manages a 20-message
ring buffer (10 turns), evicting oldest turn when full. SOUL
loaded at startup from `--soul` flag or default
`$HOME/subtract.ing/SOUL.txt`. ESC clears history.

**Why:** cal.c sent single stateless prompts with no system
context. Every query was independent — no SOUL (personality/
constraints), no memory of prior turns. The model couldn't
maintain a conversation or follow standing instructions.

**Architecture decision:** cal.c builds its own messages array
and sends directly to llama-server via HTTP, rather than
routing through hosuni.sh. This avoids modifying hosuni's
non-streaming inference path and keeps the streaming socket
approach that already works. Hosuni integration for tool use
and cloud fallback deferred to a future leg.

**Rubric:** seven binary criteria pre-committed before dispatch
(messages format, SOUL loading, history ring, history lifecycle,
non-regression, JSON escaping, buffer safety).

## Abstract

A pre-committed rubric was applied mechanically to five implementations of a session-logging extension to `handler.sh`. Two implementations tied at the top of the rubric's weighted score; the committed tiebreak rule (lowest LoC within a 2-point tie) selected a functionally valid implementation, but the tied implementation with the higher LoC produced zero valid rows when sourced into a test shell because its output included a markdown code-fence wrapper that altered shell parse semantics. An eight-model deliberative pass, dispatched across three training lineages, converged seven-of-eight on an implementation that neither the rubric nor the functional gate would have selected on their own. The finding reported here is about the three-gate methodology — static rubric, functional verification, deliberative pass — and the interactions between them.

## ./software-as-a-besides.txt (lines 1-40 of 52)
# Why subtract

subtract is software as a besides. It represents the primal translation layer native to computers.

The local model shows you the command. The lookup table pattern-matches it. Eventually you type it yourself. The AI isn't needed. That's the point.

The hope is literacy, not dependency. You graduate from "show my files" to ls, from subtract to the commands themselves. No wrapper. No subscription. No intermediary between you and the machine.

subtract shows you the supercomputer in every computer — when you look down close enough.

## Sovereignty

subtract rests on three primitives that already ship with Unix:

| Leg | Primitive | Purpose |
|-----|-----------|---------|
| Entry | command_not_found_handle | Intent enters bash |
| Trust | ssh-keygen -Y | Signed provenance for knowledge |
| Time | sleep, at, cron | Yield to the kernel scheduler |

No account. No API key. No terms of service from a foreign company. No usage meter that resets on someone else's schedule. The primitives work in any jurisdiction, on any hardware, offline.

## Energy model

As learned commands grow, the work approaches a syscall and the energy cost falls toward the kernel's own floor. Local inference stays flat per request. Cloud inference is flat or rising as orchestration accumulates. One curve goes down over time; the others don't.
