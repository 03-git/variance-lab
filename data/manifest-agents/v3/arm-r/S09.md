## hosuni deployment surfaces

There are three layers, each a different deployment surface:

---

### 1. `hosuni.c` — direct framebuffer UI (bare metal)
The interactive terminal application. From `hosuni.txt`:
> "hosuni.c talks directly to the display hardware"
- **Linux**: DRM/KMS via `/dev/dri/card*`, keyboard via `/dev/input/event*`
- **macOS**: Cocoa `NSWindow` with bitmap backing

This is the surface deployed on **acer** (Linux DRM) and **emile/rousseau** (macOS Cocoa). The user types directly into it. Inference goes over raw TCP sockets to a local llama-server/parsimony endpoint.

---

### 2. `hosuni.sh` — headless stdin/stdout daemon
From `runtime/hosuni.sh`:
> "stdin/stdout mode: read JSON request, write JSON response"
> "usage: echo '{"input":"hello","channel":"test"}' | hosuni.sh"
> "or spawned by hosuni-discord.ts per message"

This is the **inference layer abstracted away from the display**. It wraps the same tool loop (web_search, web_fetch, etc.) but with no UI — it accepts a JSON request on stdin and emits a JSON response on stdout. It talks to either parsimony pipes or a curl-based HTTP endpoint.

---

### 3. `hosuni-discord.ts` / `hosuni-discord.sh` — Discord delivery surface
From `hosuni-discord.ts`:
> "Listens for DMs and #hosuni channel messages via Gateway WebSocket"
> "Whitelist-gated auto-reply via hosuni daemon on rousseau"

This is the **Discord gateway front-end**. It:
1. Connects to Discord's Gateway WebSocket
2. Filters messages by channel (`HOSUNI_CHANNEL_IDS`) and whitelist (`~/.hosuni-discord-whitelist`)
3. Spawns `hosuni.sh` per message (the `.ts` comment says "spawned by hosuni-discord.ts per message")
4. Sends the response back via `POST /channels/{id}/messages`

The `.sh` variant (`hosuni-discord.sh`) does the same via polling REST (`POLL_INTERVAL=5`) rather than WebSocket gateway.

---

### Relationship diagram

```
Discord (remote users)
    │
hosuni-discord.ts/.sh   ← WebSocket gateway or REST poll, whitelist-gated
    │  (JSON over stdin/stdout)
hosuni.sh               ← headless inference daemon, tool loop
    │  (HTTP/TCP to llama-server or parsimony)
[inference backend]     ← llama.cpp / parsimony / Gemma-4

hosuni.c                ← direct framebuffer UI (kids/local use on acer/emile)
    │  (raw TCP sockets, same OpenAI-compat format)
[inference backend]     ← same backend
```

`hosuni.c` and `hosuni.sh` share the same **inference protocol** (OpenAI-compatible chat completions over TCP) but serve different surfaces: one is a physical display on a node, the other is a pipe endpoint for Discord delivery.
