Three parts of the codebase mirror the lookdown glob-match-then-dispatch pattern:

---

**1. `hosuni.c` — the lookdown handler itself (`lines 81-120`)**

The C implementation in `lookdown_load()` is the runtime embodiment of the pattern: read a TSV, split on tab, store `pattern` + `command` pairs, then walk the table on input. The match primitive is `fnmatch` (implied by `#include <fnmatch.h>` at line 32). First match wins — same semantics as the TSV format.

---

**2. `runtime/bin/lookdown-audit.sh`**

The audit script re-implements the same dispatch logic in bash to detect shadows:

```bash
if [[ "$cand" == $pat_j ]]; then
```

It loads entries in handler order (node-first, then universal), substitutes `*` with `x`, lowercases, and tests each later entry against all prior patterns using bash glob matching. It's a structural twin of the C handler — same ordered-table, first-match semantics, just for auditing rather than execution.

---

**3. `runtime/hosuni-discord.ts` / `hosuni.sh` (routing layer)**

The manifest lists `hosuni.sh` as the "inference launcher" and `hosuni-discord.ts` as the "discord gateway bot." Based on `lookdown.m2mini.tsv` lines like:

```
search*for*	echo "{\"input\":\"web search: ...\",\"channel\":\"terminal\"}" | ssh m1studio "bash ~/subtract.ing/runtime/hosuni.sh"
```

hosuni.sh receives a JSON envelope and dispatches by channel — `terminal`, `discord`, etc. That's the same pattern one level up: match an attribute of the input, dispatch to a handler.

---

**Structural summary:** The pattern is a stack — `lookdown.universal.tsv` and `lookdown.*.tsv` are the data layer, `hosuni.c`'s `lookdown_load`/match loop is the runtime layer, and `lookdown-audit.sh` is the verification layer. All three share ordered-table, glob-match, first-match-wins as their core invariant.
