`parsimony.c` is described in its own header as **"inference as a unix filter"** (`parsimony.c` lines 1-3).

**What it does:**

- Loads a GGUF model once, then stays alive as a long-running process
- Reads JSON requests from stdin (one per line), writes JSON responses to stdout (one per line)
- Wire format is OpenAI chat-completions-shaped: `{"messages":[{"role":"...","content":"..."}],"max_tokens":N}`
- Supports both streaming (`"stream":true`) and non-streaming responses
- KV cache clears between requests — stateless by design; the orchestrator owns conversation state

**Its role in the runtime:**

The shell manages its lifecycle via `coproc` (lines 33-37):
```sh
coproc INF { ./parsimony model.gguf; }
echo '{"messages":[...]}' >&${INF[1]}
read -r response <&${INF[0]}
```

It serves as the **local inference backend** — the T2 tier in `subtract.sh`'s routing hierarchy (local generative model, no cloud). `hosuni.c` and `subtract.sh` pipe through it to get model responses without relaunching the process per query.

The `--prelude-file` flag (e.g., `~/subtract.ing/SOUL.txt`) injects a standing system prompt at startup, which is how hosuni's disposition is loaded into the inference context before any child session begins.

In short: parsimony is the Unix-filter wrapper that makes a GGUF model behave like a line-oriented pipe — consistent with the subtract.ing primitive-first design philosophy articulated in `subtracting.html`.
