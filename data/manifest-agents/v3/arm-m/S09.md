## Three-Layer Architecture

**hosuni.c** is the framebuffer computing surface — DRM/evdev, keyboard input, direct display output. It's the local, physical deployment: a machine with a screen and keyboard, no network. Manifest tags it `[runtime]` alongside `platform_linux.c` and `hosuni.h`. It's the primitive layer.

**hosuni.sh** is the inference daemon layer. From lines 1-9:
> "listens on a port via nc. receives JSON, checks lookdown, routes inference through parsimony (coproc via named pipes), parses tool calls, executes, loops, logs."

It sits between callers and the local llama.cpp inference stack (`parsimony`). It handles tool execution, YouTube transcript pre-fetch, web search pre-fetch (lines 296–318), conversation state, and SOUL.txt injection. It is network-addressable (nc on a port) but local to the node.

**hosuni-discord.ts** (and `hosuni-discord.sh`) is the remote gateway layer. It bridges Discord's WebSocket Gateway API into hosuni.sh calls. The `.ts` file is a Deno bot; the `.sh` file is a polling fallback. Both are whitelist-gated and log to `~/human/sessions/hosuni-discord.log`. The two channel IDs (`1489355263361286204`, `1483978150852559030`) are hardcoded in both.

## Deployment Surfaces

| Surface | Entry point | Transport | Human present? |
|---|---|---|---|
| Physical kiosk | `hosuni.c` | DRM framebuffer + evdev keyboard | Yes, at the screen |
| Local daemon | `hosuni.sh` | `nc` on a port (JSON over TCP) | Indirect |
| Discord DM/channel | `hosuni-discord.ts` or `.sh` | Discord Gateway WebSocket / REST poll | Remote, async |

The dependency chain runs one way: Discord → hosuni.sh → parsimony/llama.cpp. `hosuni.c` is a parallel surface that does not route through `hosuni.sh` — it is the framebuffer frontend, and the memory index (`project_cal_hosuni_integration.md`) records that wiring hosuni.c → hosuni was a separate integration workstream, not the original design.
