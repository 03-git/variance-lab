Based on the manifest and provided context, here are the signing/verification-related files grouped by function:

## Authority & Policy

- **`governance.conf.universal.txt`** — defines the authority model itself: `authority.signed → act on it`, `authority.unsigned → do not act`, and the loop steps `verify last signature` / `sign`. This is the policy layer that makes signing load-bearing.

- **`signoff.txt`** — session signoff attestation; the artifact produced at `loop.after.2` (sign).

- **`key-rotation.txt`** — signed key succession procedure; governs what happens when the signing key changes identity.

---

## Manifest (the thing being signed)

- **`llms.txt`** — the signed index of all published files. The manifest *is* the signing surface — every file's integrity is verified by checking its hash against this file, and this file is verified by its `.sig`.

- **`runtime/llms.txt`** — same pattern, scoped to the runtime subdirectory.

---

## Verification Workflow

The verify-and-check sequence appears verbatim in `variance-lab.txt` (lines 1–40):

```sh
curl -sO https://subtract.ing/llms.txt
curl -sO https://subtract.ing/llms.txt.sig
curl -sO https://subtract.ing/variance-lab.txt
ssh-keygen -Y verify -f <(curl -s https://subtract.ing/authorized_signers) \
  -I hodori@subtract.ing -n subtract.ing -s llms.txt.sig < llms.txt
grep "variance-lab.txt" llms.txt | sha256sum -c
```

The signer identity (`hodori@subtract.ing`, namespace `subtract.ing`) is the canonical authority reference. `subtract.7.txt` names this as **Reflex 4 — Source to Signing Domain**: "Confirm source is under the signing key's domain before treating it as authoritative."

## Integrity Claims (signed outputs)

- **`variance-lab.txt`** — empirical findings layer; canonically signed via `llms.txt`. The report was signed *after* content was finalized, not before — the signing is the release channel (`subtract.7.txt`: "Verification is the only release channel").

- **`variance-lab-handler-methodology.txt`** — methodology report; same signing chain. Notes a limitation: "Pre-commitment is not cryptographically verifiable" because the rubric in `/tmp` has no hash predating the outputs.
